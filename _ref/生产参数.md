## §1 封面

### 视觉系统

| 参数 | 值 | 说明 |
|------|-----|------|
| 画布 | 1920×1080 | B站 16:9 |
| 底图 | 视频截图，主讲人正脸 | 1920×1080 LANCZOS 缩放 |
| 底图亮度 | 0.80 | 即"透明度 80%"，微微压暗（不是 80% 不透明！） |
| 主字体 | SimHei（黑体） | `C:\Windows\Fonts\simhei.ttf`，系统最粗中文 |
| 副字体 | Segoe UI Emoji | 底部信息条用，`C:\Windows\Fonts\seguiemj.ttf` |
| 主色 | 暖黄 `#FFC82D` | 标题和装饰线 |
| 辅色 | 暖白 `#FCFAF5` | 副标题 |
| 填充 | 纯黑 `#000` 2px | 四周 8 方向填充，模拟超粗 |
| 装饰线 | 120×4px | 主副标题之间，居中 |
| JPG 质量 | 92 | 超 5MB 自动降至 75 |
| 底部信息条 | 色值 `(200,200,200)` | 灰色小字，"YouTube · <来源> | 猫波译站" |

### 布局规则

```
┌─────────────────────────────────────────────┐
│                                             │
│              [底图，亮度 0.80]               │
│                                             │
│          副标题（62px，暖白）                │
│          主标题（165px，暖黄）               │  ← 竖向居中，偏上 60px
│        ────── 装饰线（120px）────────        │
│          出处行（28px，暖白）                 │
│                                             │
│     YouTube · Sequoia Capital  |  猫波译站    │  ← 底部 30px margin
└─────────────────────────────────────────────┘
```

- 全部居中（`anchor="mm"`）
- 行间距：标题后 40px，其他 30px
- 装饰线在主/副标题之间（两行都有时才显示）
- 最多 3 行文字 + 1 条线 + 1 条底栏
- **不加频道水印**

### 文字排版约束

- **主标题（title）**：165px SimHei，暖黄，≤15 字。最核心论断
- **副标题（sub）**：62px SimHei，暖白，≤20 字。补充/反直觉钩子
- **出处行（source）**：28px SimHei，暖白。如 "AI Ascent 2026 · Sequoia Capital"
- 三行可独立存在（任意行缺省不报错），2 行测常见配置
- 标题文字从 transcript 中提取，人工判断后填入 `--title` `--sub` 参数

### 截图选帧规则

1. `ffmpeg -i video.mp4 -ss <timestamp> -vframes 1 frames/frame_XXXXs.jpg`，至少 5 个时间点
2. 选帧标准：
   - 主讲人正面，面部清晰（光照好、无遮挡）
   - 背景干净（不杂乱、不与文字重叠区冲突）
   - 表情自然（不眨眼、不张嘴一半）
3. 选一张最好的保留，其余删除
4. 时间点均匀分布（每隔视频长度/5 截一张），但优先选中间段（主讲人状态最好）

### 设计决策来源

- 底图亮度 0.80：用户纠正——"透明度 80%"指透过程度，不是不透明度 80%。见 `_ref/pitfalls.md` §3
- SimHei 字体选择：PIL 不支持可变字体轴参数，NotoSansSC-VF 只拿 Regular。见 `_ref/pitfalls.md` §5
- 四周填充 2px：替代描边，165px 字号下描边 >3px 会糊
- 无频道水印：频道视角—封面是内容的第一触点，水印分散注意力

## §2 标题

### 硬约束

- **≤80 字**（B站标题字数上限）
- **不含「双语字幕」**（占字数，用户看封面和标签就知道）
- **文件名 = 标题**（B站上传时自动识别填入）

### 标题公式

```
<嘉宾身份> <嘉宾名>：<核心论断>
```

| 部分 | 作用 | 字数 | 示例 |
|------|------|------|------|
| 嘉宾身份 | 人物锚点，建立可信度 | 4-10 字 | "Claude Code 之父" |
| 嘉宾名 | 可搜索的关键词 | 英文原名 | "Boris Cherny" |
| 核心论断 | 最独特/反直觉的一句话 | 15-40 字 | "编程已解决，接下来是什么" |

变化体（嘉宾知名度足够时）：

```
<嘉宾名>：<核心论断> — <补充钩子>
```

### 核心论断提取法

从 transcript 中找符合以下条件的句子：

1. **反直觉**：挑战常识的断言
2. **可量化**：有具体数字/时间的预测
3. **独特性**：只有这个人会说/能说的话
4. **情感张力**：令人惊讶/兴奋/担忧的判断

**排除项：** 通用描述、嘉宾没说的话、过于技术化的术语。

### B站标题规律

- **断言式 > 疑问式**
- **具体 > 抽象**：有数字/场景的标题比空泛概念好
- **人物前置**：嘉宾名字放在标题前三分之一
- **避免标题党**：论断必须来自视频内容
- **标点克制**：不用问号三连、不用感叹号刷屏

### 不可用模式

| 模式 | 为什么不用 |
|------|-----------|
| `《》书名号` | B站标题不流行书名号 |
| `【】方括号` | 搬运号常用，降低点击欲 |
| `#话题标签` | 标题不加 hashtag，放简介里 |
| `「」引号` | 日式引号，科技区不合群 |
| 表情符号 emoji | 科技区标题保持严肃感 |
| 全角问号/感叹号堆叠 | 低质号特征 |

### 标签规则

- ≤10 个标签，每个 ≤12 字
- 优先级：人物名 > 技术关键词 > 领域分类 > 来源/活动
- 示例：`Claude Code` `Anthropic` `AI编程` `Boris Cherny` `Sequoia` `AI Ascent` `编程未来` `开发者工具` `大模型` `人工智能`

## §3 字幕

### ASS 技术参数

| 参数 | 值 | 说明 |
|------|-----|------|
| 画布 | 1920×1080 | ASS PlayResX/Y |
| 中文字体 | SimHei 42px | 无粗体（SimHei 原生够粗） |
| 英文字体 | Segoe UI 32px Bold | `\b1` 加粗 |
| 中文位置 | `\N` 前（上半部分） | 视觉优先级中文 > 英文 |
| 英文位置 | `\N` 后（下半部分） | |
| MarginV | 45 | 底部边距 |
| Alignment | 2 | 底部居中 |
| 换行 | `\N` 硬换行 | ASS 单事件内部分行 |
| WrapStyle | 2 | 智能换行（按字符，下粗上细） |

### ASS 格式（唯一标准）

```ass
[Script Info]
Title: Bilingual Subtitles
ScriptType: v4.00+
PlayResX: 1920
PlayResY: 1080
WrapStyle: 2

[V4+ Styles]
Style: Default,SimHei,42,&H00FFFFFF&,&H00000000&,&H00000000&,&H00000000&,0,0,0,0,100,100,0,0,1,2,0,2,10,10,45,1

[Events]
Format: Layer, Start, End, Style, Name, MarginL, MarginR, MarginV, Effect, Text
Dialogue: 0,0:00:01.00,0:00:04.00,Default,,0,0,45,,{\fnSimHei\fs42}中文\N{\fnSegoe UI\fs32\b1}English
```

核心规则：**一条 Dialogue = 一个字幕事件**。用 `\N` 分隔中英文，不用两条 Dialogue。

### 单事件 vs 双事件（为什么不能拆）

libass 有碰撞检测（collision detection）：两条相邻 Dialogue 时间重叠且位置接近时，自动隐藏其中一条。双 Dialogue 方案会导致中英文随机消失一个。单事件 `\N` 方案是唯一正确解。见 `_ref/pitfalls.md` §4。

### 中文字数控制（≤28 字）

- 中文超过 28 字 → 在中文标点处拆分（`，。！？；、`）
- 按字数比例分配时间戳
- 无标点长句超过 56 字（max_chars × 2）→ 硬切

```python
enforce_max_chars(srt_path, max_chars=28)  # 注意签名：不是 (srt_path, output_path)！
```

Boris Cherny 40 分钟演讲验证（~300 条字幕 → 拆分后 ~350 条）。

**为什么是 28 字**：B站观众中文阅读 ~4 字/秒，字幕停留 3-5 秒 → 12-20 字舒适。技术内容需要更长上下文 → 28 字上限。28 字 = 42px SimHei 在 1920px 宽屏约 2 行。

### 翻译质量控制

**Prompt：**

```
System: You are a professional EN→ZH subtitle translator. Output ONLY the Chinese translation for each line, one per line. No explanations.
User: Translate these English subtitles to Chinese (Simplified). For each input line, output ONLY the Chinese translation on a single line. Keep it concise and natural. Preserve technical terms where appropriate.
```

**关键参数：**

| 参数 | 值 | 原因 |
|------|-----|------|
| model | deepseek-chat | 性价比最高 |
| temperature | 0.3 | 字幕需要一致性，不要创意 |
| 批大小 | 10 条/批 | 平衡速度与上下文 |
| 并发 | 5 | DeepSeek 免费版 ~5 QPS 限流 |
| 超时 | 120s | 单批翻译 |
| 重试 | 3 次，指数退避 1s/2s/4s | 网络抖动 |

**行数对齐**：API 返回行数必须与输入一致。多了截断，少了补空行。空行保留不跳过——字幕 index 对齐靠行位置。

### 断句规则

- 目标 ~18 英文词/句
- 句尾标点（`. ! ?`）触发切割
- 超 27 词（18 × 1.5）无标点也切割
- carryover 裁剪：只裁文本，不移时间戳（避免字幕闪烁）
- 输出：`02_seg.srt`

### 字幕同步

当前不做全局偏移修正。YouTube 自动字幕可能存在 ~200ms 延迟，目前可接受。如需修正：pipeline.py 可加 time shift 参数，全局 +/- N ms。

### 已知限制

- 不处理多人对话（DeepSeek 翻译无法区分说话人）
- 不处理屏幕文字（如 PPT 上的英文）
- 演讲者语速过快时（>200 wpm），28 字限制会导致拆分过碎
- 技术术语（如 "agentic coding"）首次出现时翻译可能不统一

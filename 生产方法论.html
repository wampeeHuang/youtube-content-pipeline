00任务书：项目定义 · 张咋啦

任务书定义完整项目需求——做什么、交付什么、怎么管、怎么判断做对了。后续管线全部服务于这份任务书。

0.1 项目动机

为 B站科技区观众提供高质量中文字幕的海外 AI 访谈搬运。消除语言障碍，保留原汁原味——不做 AI 配音、不做二次创作、不做观点嫁接。

定位：AI 辅助 + 人工品控的精翻搬运。差异化：精翻品质（非机翻）、精选嘉宾（非新闻批量）、完整管线可复现、封面/标题适配 B站审美。

0.3 管线全流程（需求层级）

管线是满足任务书的手段——任务书回答"要什么"，管线回答"怎么做"。分两段：机械段（②-⑧ 独立阶段脚本，每阶段可单独重跑）+ AI 段（⑧-⑬ 每步需 Claude 判断）。完整实现见 §03。

① 选题→ ② 下载→ ③ 断句→ ④ 翻译→ ⑤ 判定→ ⑥ ASS→ ⑦ 渲染→ ⑧ 金句提取→ ⑨ 封面→ ⑩ 标题→ ⑪ 元数据→ ⑫ 专栏→ ⑬ 发布

关键关口：④ 翻译是素材质量判定点——翻译完成后才能判断内容价值，决定继续或排除（见 §01 影视飓风差异）。⑤ 判定是 Go/No-Go 决策点。⑧ 金句提取决定封面和标题的钩子质量。

排除标准（一票否决）

#条件原因
1B站已有精品中字 且 内容为纯新闻/时效型（≤1 个月有效期）需求已被满足 + 无长尾价值
2版权敏感（电影/音乐为主）合规红线
3纯娱乐 / 标题党 / 营销号风格与频道定位不符
4语音质量差——口音过重/背景噪音大/多说话人重叠/远场录音ASR 准确率 < 85%，翻译质量不可控
5翻译后主观判定搬运价值不足Cat Wu Tokyo 2026 先例
6纯 talker——嘉宾无建造/研究记录，仅观点输出信息密度低，观点缺乏实践验证
7内容为纯新闻/时效型（≤1 周有效）与长期价值定位冲突

信息密度（操作化定义）

信息密度不是感觉，是可计量的比率。初筛快速估算，翻译后精算：

指标算法高密度低密度（排除）
观点密度独特观点数 ÷ 视频时长（分钟）≥ 2 个/分< 0.5 个/分
数据引述率数字/百分比/时间跨度的出现频率≥ 1 次/3 分钟无数据仅观点
实操占比包含具体步骤/方法/工具的句段 ÷ 总句段≥ 30%< 10%
语音填充率非信息填充词（um/you know/我觉得）占比< 15%> 30%（访谈松垮信号）

翻译后精算：随机抽 3 段 × 2 分钟人工评估，与估算值偏差 > 30% 时以精算为准。AI 辅助自动评分待建设。

0.8 发布节奏

更新频率

日均 2 条，上限 3 条。AI 管线日产能力远超此数，约束不在人力在算法——同频道日发 3 条以上内部抢量，边际播放递减。数据依据见 §5.3 投稿频率限制测绘。

周均 12-15 条。周末流量不衰减（知识区用户周末深度观看时长更长），照常更新不降频。

发布时间窗

推荐时点：10:00 / 14:00 / 18:00。间隔 ≥ 2 小时，给每条留出算法消化窗口。

B站风控下限 12-15 分钟间隔，但 30 分钟内连发触发隐性降权。规划时距 ≥ 2 小时。

单日管线流

动作窗口产出
扫源 + 评分09:00-10:00候选池新增 + 排期决定
批量触发翻译管线（多点并行）10:00-12:00ASR + 翻译 + ASS（2-3 条同时跑）
审校 + 封装 + 封面13:00-16:00终版字幕 + 成片 + cover.jpg
分批发布10:00 / 14:00 / 18:00B站上传 + 信息填写
复盘（次日）09:0024h 数据记录

管线一条从选题到发布标准跨度 12-24h（AI 并行缩短了人力串行瓶颈）。当天选题当天发是常态。

发稿节奏优先级

匀速日更 > 囤积爆发。宁可今天发 1 条保节奏，不等 3 天后一天发 4 条。突发大事件（GPT-6 发布会等）→ 优先插队，常规排期顺延。

缓冲机制

维持 3-5 条备选稿（已完成翻译审校，待发布）。周末不生产，备选稿覆盖。囤稿 ≤ 5 条，超限停止生产先消化。管线中断时备选稿保证当天不断更。

01案例分析 · 潘乱

三个正面案例 + 一个反面案例。每个案例分析五个维度：可借鉴的结构、不借鉴的原因、利益结构、分发逻辑、不可替代性。正面案例告诉我们怎么做，反面案例告诉我们为什么不做——两者同等重要。

1.1 影视飓风：原创方法论 vs 搬运的差异

B站 900 万+粉丝，科技区标杆。2025 年首次公开十年创作方法论。

壳理论（对我们最重要的一条）

影视飓风：先定标题和封面，再定内容架构——用生活化钩子（壳）包裹硬核知识（核）。
例：《泰勒演唱会为什么没有摩尔纹？》400 万+播放 vs"什么是摩尔纹"无人问津。

搬运的差异：我们内容已存在，不能"先定标题再拍"。流程是反过来——先翻译才能判断素材质量。
可借鉴：翻译完成后，从 transcript 中提取最精彩的金句作为"壳"，倒推设计封面和标题。壳不是编的，是从已有内容中选的。Kevin Weil 期用的就是这套——"你现在用的 AI 是你余生最差的"= 壳，Kevin Weil 的完整访谈 = 核。

HKRR 四维 → 我们的质量门禁

维度影视飓风我们的适配
Happiness 快乐让观众感到愉悦搬运不创造快乐，但可以选择信息密度高、不枯燥的内容
Knowledge 知识提供信息增量核心维度。翻译后判定：有没有新观点/新数据/新框架？不过关 → 排除
Resonance 共鸣情绪层面的连接嘉宾的坦诚度、反直觉观点、行业内幕 → 观众"原来如此"感
Rhythm 节奏信息密度的控制赞助商口播 = 节奏断裂点，需检测并跳过。断句参数控制阅读节奏

1.2 AI译片君：赛道参照

AI 翻译+语音合成，批量产出，27 万粉 / 3.2 亿播放。B站 AI 搬运赛道最大玩家。

借鉴什么

  • 深色封面 + 暖色大字 — 已验证的 B站科技区最优配色
  • 稳定更新频率 — 习惯性订阅靠的是产出节奏
  • 标题格式化 — 嘉宾身份 + 核心观点，清晰可预期

不借鉴什么

  • AI 配音覆盖原声 — 我们去掉了原汁原味的最大优势
  • 批量机翻品质感低 — 观众能分辨，不挂"双语字幕"标签的原因
  • 早期同类号已被处罚 — 合规风险：转载标注必须清晰
  • 新闻搬运长期价值低 — 不做日更/纯新闻

赞助商检测（Karpathy 方案 · 2026-06-20）

背景：Kevin Weil 期 167/247 条 (68%) 翻译失败。根因不在翻译 prompt——是把"判定是否值得翻"和"翻译"混在了同一个步骤里。DeepSeek 遇到 ASR 质量差的英文直接跳过整行。D-02 已判：赞助商段直接剪切，不走翻译。

核心思路：赞助商检测是一个 LLM 擅长的二分类问题（文本模式匹配），不需要训练专门模型。

环节内容
触发时机断句 (③) 之后、翻译 (⑤) 之前
输入02_seg.srt
输出02_seg_clean.srt (去赞助商) + _sponsor_cuts.json (ffmpeg 剪切点)
工具DeepSeek API, temp=0, batch=20, 并发=5
耗时~30s (247 条 → 13 批)
人工复核扫一眼标记结果 (~5m)，防 false positive 误杀

检测 Prompt

System: You are a content classifier. For each subtitle segment below,
  answer ONLY "yes" or "no" — is this segment part of a sponsor/ad read?
  Sponsor indicators: brand names repeated, discount codes, "thanks to our
  sponsors", "check out", "use code", fast speech artifacts (word repetition
  in ASR). Answer one word per line, exactly matching the input line count.

  User: [N 条 SRT 文本，每行一条]

检测后处理

  1. 标记为 "yes" 的段落 → 从字幕中移除
  2. 相邻被切段落的时间间隙 > 0.5s → 写入 _sponsor_cuts.json
  3. 翻译用 clean SRT（无赞助商段），保证 100% 翻译覆盖率
  4. 渲染时 ffmpeg 按 _sponsor_cuts.json 剪切视频
  5. 人工复核：确认无 false positive（非赞助内容被误标）
March of Nines 警告：唯一不能接受的是 false positive——把嘉宾正片当赞助商切掉。这是不可逆的信息损失。所以 temp=0（不创造），人工确认切点（Iron Man suit，不是 Iron Man robot）。false negative（漏切赞助商）只是多几秒广告，不影响内容质量。
代码量目标：~60 行 Python。一个函数读 SRT → 分批送 API → 返回标记列表 → 写两份输出。Don't be a hero.

问题三：封面视觉效果不足

症状分类

症状用户会看到影响
A. 字体太瘦 + 有黑边标题文字纤细，外圈有黑线，像 Word 艺术字不够醒目，不像科技区封面
B. 底图无吸引力背景是纯代码/文档截图，没有人物、没有光影点击率低于预期，没有"想点进去看"的冲动

根因分析

症状直接原因技术细节
A. 字体瘦+黑边SimHei（黑体）本身非粗体 + _draw_bold_text() 使用黑色 perimeter fill 模拟粗体perimeter fill = 在文字周围画 N 层半透明黑色 → 视觉上字体瘦且带黑晕
B. 底图差帧选择无视觉标准，默认取中间帧，落到哪帧算哪帧没有"是否有人脸""是否有代码"的判断，帧选择 = 随机

标准修复

问题修复效果
A. 字体改用 msyhbd.ttc（微软雅黑 Bold），draw.text() 直接纯色绘制，无 perimeter fill字体粗壮有力，纯色边缘清爽，科技感强
A. 字体（备选）msyhbd.ttc 不可用时回退 simhei.ttf不因字体缺失导致封面生成失败
B. 底图帧选择增加视觉判断标准（待定案）提高封面底图质量，进而提高点击率

字体加载实现

FONT_BOLD = "msyhbd.ttc"      # 微软雅黑 Bold — 粗体无描边
FONT_FALLBACK = "simhei.ttf"  # 备选黑体

def _load_font(size: int, bold: bool = True) -> ImageFont.FreeTypeFont:
    candidates = [FONT_BOLD, FONT_FALLBACK] if bold else [FONT_THIN]
    for name in candidates:
        path = _find_font(name)
        if path:
            try:
                return ImageFont.truetype(path, size)
            except Exception:
                continue
    return ImageFont.load_default()

关键改动：移除 _draw_bold_text() 和 DARK_OVERLAY 常量，改为 _load_font() + 直接 draw.text() 纯色绘制。

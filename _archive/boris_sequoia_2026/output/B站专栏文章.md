# 编程已解决：Claude Code 之父 Boris Cherny 论 AI 编程的「印刷术时刻」

> 红杉资本 AI Ascent 2026 · 40 分钟深度对话 · 双语字幕版已发布
> 嘉宾：Boris Cherny（Claude Code 创建者 & 负责人，Anthropic）
> 主持：Lauren Reeder（红杉资本合伙人）
> 原视频：https://www.youtube.com/watch?v=SlGRN8jh2RI

---

编程，作为一门专业技能，已经被 AI 解决了。

这不是标题党。说这话的人是 Boris Cherny——Claude Code 的创建者。他自 2025 年 11 月起没有手写过一行代码，每天靠 AI 提交 10 到 30 个 PR，最高纪录一天 150 个。他用 iPhone 管理着数百个并行 AI Agent，他的团队里设计师、数据科学家、财务、用户研究员——每个人都在写代码。

在红杉资本 2026 年 AI Ascent 大会上，Boris 抛出了一个历史级类比：**AI 对编程的民主化，正如同古腾堡印刷术对识字的民主化。**

以下是这场 40 分钟对话的核心内容。

---

## 一、Claude Code 是怎么来的？一个意外

"我加入 Anthropic 是在 2024 年底。那时候公司内部有一个孵化器叫 Labs。我们团队很小，就几个人。我们创建了 Claude Code、MCP 和桌面应用。然后团队就解散了。"

Boris 说，当时的驱动力是"产品悬垂"（Product Overhang）——**模型能做的事，远超出了产品形态能承载的范围。**

"2024 年底看编程工具，最先进的还只是行级自动补全。你打开 IDE，按 Tab，它帮你补一行代码。但 Sonnet 3.5 让我们觉得，可以走得更远。不用自动补全了，直接让 Agent 写代码。"

于是他动手做了。然后——**前 6 个月完全不行。**

"真的不好用。几乎没法用。我自己写代码，它大概只能顶 10%。甚至发布之后，也没有立刻火起来。"

真正的拐点是 2025 年 5 月，Opus 4 发布。"我记得非常清楚，就是从那一刻开始，指数增长。然后每一个模型版本——4.5、4.6、4.7——每一次都再上一个台阶。"

**他在产品市场匹配（PMF）到来之前，就确信它会来。** 因为他知道下一个模型能做到什么。

---

## 二、"编程已解决"：100% AI 代劳

Boris 在现场做了一个调查："在座谁 100% 手写代码？"几乎没人举手。"谁 100% 用 Agent 写代码？"零星几只。

"对我来说是 100%。Claude Code 自己的代码库——你们知道的，它泄露了——其实很简单，就是 TypeScript 和 React。选这个技术栈是因为模型对它的分布最好。放在今天，什么语言都无所谓了，模型没见过的新语言也能直接上手。"

"我们大概在 2025 年 10 月、11 月左右，达到了模型写 100% 代码的程度。从那以后，我每天提交十几个 PR。上周有一天我搞了 150 个——就想看看极限在哪。"

但他也坦诚：**"这不适用于所有地方。** 有些非常庞大的代码库、一些奇怪的编程语言，模型还不擅长。这些地方，通常的答案是——等下一个模型。"

---

## 三、一个人的开发舰队：手机 + 数千 Agent

Boris 分享了他当前的个人工作流，让现场开发者目瞪口呆：

**"我现在大部分工作是在手机上完成的。"**

他的 iPhone 上开着 Claude 应用，里面有 5 到 10 个会话。每个会话里有一堆 Agent——"现在大概有几百个在跑。每天晚上，我大概有几千个在做更深的工作。"

管理这么多 Agent 的核心工具是一个他称为"最酷的东西"的功能：

**Loops（循环）**——用 Claude 设置 cron 定时任务，让 Agent 在未来的某个时间点自动执行并重复。

"我现在有几十个循环在跑。一个在盯着我的 PR，自动修复 CI、自动 rebase。另一个在维护 CI 健康——如果有不稳定的测试，它会自己去修。还有一个每 30 分钟从 Twitter 抓取用户反馈、帮我聚类整理。"

"任何时间点都有一堆循环在运行。**我觉得循环就是未来。** 如果你还没试过，强烈推荐。我们还推出了 Routines（例程），跟循环一样但跑在服务器上——你把笔记本合上，它继续跑。"

---

## 四、未来的团队：跨学科通才

"我觉得未来的趋势是，通才会比现在多得多。但现在的"通才"通常还是工程师——就是写代码的，只不过 iOS 也写、后端也写。这是工程内部的通才。"

**"但我们会看到越来越多跨学科的通才。** 就是那些擅长产品工程、同时又很会做设计的工程师。或者同时懂数据科学和工程的人。"

"在 Claude Code 团队里，**每个人都在写代码。** 工程经理、产品设计师、数据科学家、财务、用户研究员——每个人。他们是某个领域的专家，但现在每个人同时也在编程。"

Boris 认为，这个趋势不只适用于 Anthropic："我注意到台下有人在点头。我打赌在座各位并不觉得这很惊讶，因为你们也在经历同样的事。"

---

## 五、SaaS 的末日？护城河重估

当主持人问"AI 让写代码便宜了 100 倍，软件产品的价值会怎么样？会有 SaaS 末日吗？"

Boris 引用了播客 *Acquired* 中 Hamilton Helmer 的"七种力量"（7 Powers）商业护城河框架：

**"AI 会让某些护城河变得不那么重要。"**

- **转换成本**：模型可以轻松地把你的系统从 A 迁移到 B。
- **流程优势**：Claude 4.7 可以针对任何目标自动爬山优化。你给它一个目标，它会迭代到完成为止。"这可能是第一个能做到这样的模型。"

**"但之前的其他护城河仍然有效。** 网络效应、规模经济、稀缺资源——这些不会因为 AI 而改变。"

**"如果你现在从零开始创业，你可以从第一天就用 AI 原生地构建一切。** 大公司必须改造业务流程、重新培训所有人使用新技术、面对巨大的内部阻力——而你没有这些问题。"

"我不知道具体会怎么样。**但我认为这是最好的创业时代。有史以来最多的颠覆正在发生。**"

---

## 六、印刷术时刻：编程将成为全民技能

当被问到"编程会变成像 Microsoft Office 一样的通用技能吗？"Boris 的回答格外笃定：

"哦，天哪，会的。不止于此。就像，我不知道，发短信一样。"

他给出了整场对话中最具启发性的历史类比：

**"15 世纪的欧洲，只有 10% 的人识字。识字的人通常受雇于不识字的国王和领主——读和写是一项专业技能。然后印刷术被发明了。在第一批文学作品出版后的 50 年里，欧洲出版的书籍数量超过了此前一千年。"**

"同一时期，一本书的成本下降了大约 100 倍。接下来几百年间——因为学读写不容易，需要教育体系、需要政府推动——但全球识字率从 10% 上升到了 70% 以上。现在，我们都能读能写，不需要专门为此拿一个学位。虽然专业作家这个职业依然存在。"

**"我认为同样的事情即将在软件领域发生，而且会比 50 年快得多。软件将被完全民主化。任何人都可以做。"**

"举个具体例子：**写会计软件最好的人，可能今天就已经不是工程师了，而是一个真正优秀的会计师。** 因为他们非常了解业务领域，而写代码反而是简单的部分。难的是理解那个领域。"

---

## 七、模型 vs 产品：什么更重要？

现场有人问：Claude Code 的成功，多大程度上归功于模型能力，多大程度归功于产品设计决策？

Boris 的回答很坦诚：

"一年前可能是五五开。但**随着模型越来越好，产品'外壳'的重要性在降低。**"

"我们思考的是如何进化这个外壳——让循环成为更一等公民的功能，让跑大量 Agent 变得更容易。子 Agent 是一个思路，还有很多在开发中。"

**"一年后，模型对齐会好得多。** 今天我们围绕提示注入、静态验证、命令权限模式、人机循环做的所有安全机制，都会变得不那么重要。因为模型自己会做对的事。"

---

## 八、Anthropic 内部：全公司零手写代码

"如果你去 Anthropic 跟人聊聊，你会发现我们用 Claude 做一切事情。Claude 之间全天候在交流——我在写代码的时候，我的若干个循环 Agent 会通过 Slack 跟其他人的 Agent 沟通，一起搞清楚一些问题的答案。"

**"公司里没有任何地方还有手动编写的代码了。** 所有 SQL 都是模型生成的。一切都由 AI 构建。"

"所以我认为，我们领先的地方不是技术——用的是跟外面一样的模型——**真正更领先的是组织结构和流程。** 因为我们在打造一个平台，开发者能用我们自己在用的东西至关重要。但如何围绕 AI 重组团队和流程，这才是有更大提升空间的地方。"

---

## 最后

当被问到"未来 6 到 12 个月，模型变强之后，什么东西会变得特别有价值？"

Boris 的答案很干脆：**"Claude Design 就是一个很好的例子——现在已经不错了，接下来会好很多。Claude Code 还有几个即将上线的东西。循环、批量处理、大规模并行 Agent——这些都会更好。Computer Use 也是。"**

他没有预告一个具体的时间线。因为 Anthropic 的规划方式是："我们只规划未来一周的事。"

但方向很明确：**模型正在从工具变成环境。未来的工程师不需要决定用什么模型、跑在本地还是云端——这些决策将由 Agent 自己来做。我们要做的，是建造值得让 Agent 去建造的东西。**
